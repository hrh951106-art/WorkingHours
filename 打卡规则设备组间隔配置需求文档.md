# 打卡规则设备组间隔配置需求文档

## 文档信息
- **文档版本**：v1.0
- **创建日期**：2026-04-08
- **需求类型**：功能新增
- **优先级**：中
- **状态**：待评审

---

## 一、需求背景

### 1.1 当前现状
目前打卡规则（PunchRule）仅支持全局的打卡间隔配置，通过 `beforeShiftMins` 和 `afterShiftMins` 字段来设置班次前后的打卡有效时间。该配置对所有设备和所有员工统一生效，无法针对特定设备组进行差异化配置。

### 1.2 业务痛点
在实际业务场景中，存在以下痛点：
- 不同车间/区域的设备组可能需要不同的打卡间隔配置
- 某些特殊设备组可能需要更宽松或更严格的打卡时间限制
- 无法灵活针对特定设备组制定个性化的打卡规则

### 1.3 需求目标
支持在打卡规则中针对不同设备组配置独立的刷卡间隔，提高打卡规则的灵活性和可配置性。

---

## 二、需求概述

### 2.1 需求描述
打卡规则新增时需要支持配置间隔功能，具体要求如下：

1. **支持针对设备组配置刷卡间隔**：可以为特定设备组设置独立的打卡间隔时间
2. **支持配置多套间隔**：一个打卡规则可以配置多个设备组的间隔
3. **设备组非必选**：可以不选择设备组，此时使用默认的全局间隔配置
4. **刷卡间隔非必选**：可以选择设备组但不配置间隔，此时仍使用全局配置
5. **唯一性约束**：在同一个打卡规则中，一个设备组不能配置两个刷卡间隔

### 2.2 核心业务规则
- 打卡规则的优先级（priority）仍然生效
- 如果员工打卡时涉及多个匹配的规则，按优先级从高到低匹配
- 对于设备组间隔配置，优先使用设备组特定的间隔，如果未配置则使用全局间隔
- 同一设备组在同一个打卡规则中只能配置一次

## 三、功能详细说明

### 3.1 功能模块
本需求涉及以下功能模块：

#### 3.1.1 打卡规则管理模块
- **新增**：在创建打卡规则时，可以添加设备组间隔配置
- **编辑**：在编辑打卡规则时，可以修改或删除设备组间隔配置
- **查看**：在打卡规则详情中，展示设备组间隔配置列表

#### 3.1.2 设备组管理模块
- **查询**：支持选择已创建的设备组
- **展示**：在打卡规则中显示设备组名称和代码

### 3.2 数据模型设计

#### 3.2.1 方案一：扩展 PunchRule.configs JSON 字段
利用现有的 `configs` 字段（JSON格式）存储设备组间隔配置。

**JSON 结构示例：**
```json
{
  "deviceGroupIntervals": [
    {
      "deviceGroupId": 1,
      "deviceGroupCode": "GROUP_A",
      "deviceGroupName": "车间A设备组",
      "beforeShiftMins": 30,
      "afterShiftMins": 30
    },
    {
      "deviceGroupId": 2,
      "deviceGroupCode": "GROUP_B",
      "deviceGroupName": "车间B设备组",
      "beforeShiftMins": 60,
      "afterShiftMins": 60
    }
  ]
}
```

**优点：**
- 无需修改数据库结构
- 灵活性高，易于扩展

**缺点：**
- 查询和索引相对复杂
- 数据校验需要应用层处理

#### 3.2.2 方案二：创建关联表（推荐）
创建新的关联表 `PunchRuleDeviceGroupInterval` 来存储设备组间隔配置。

**表结构设计：**
```prisma
model PunchRuleDeviceGroupInterval {
  id               Int         @id @default(autoincrement())
  punchRuleId      Int
  deviceGroupId    Int
  deviceGroupCode  String
  deviceGroupName  String
  beforeShiftMins  Int         @default(0)
  afterShiftMins   Int         @default(0)
  status           String      @default("ACTIVE")
  createdAt        DateTime    @default(now())
  updatedAt        DateTime    @updatedAt
  punchRule        PunchRule   @relation(fields: [punchRuleId], references: [id], onDelete: Cascade)
  deviceGroup      DeviceGroup @relation(fields: [deviceGroupId], references: [id])

  @@unique([punchRuleId, deviceGroupId])
  @@index([punchRuleId])
  @@index([deviceGroupId])
}
```

**需要修改的模型：**
```prisma
model PunchRule {
  id                          Int                          @id @default(autoincrement())
  code                        String                       @unique
  name                        String
  priority                    Int
  conditions                  String
  beforeShiftMins             Int                          @default(0)
  afterShiftMins              Int                          @default(0)
  configs                     String                       @default("[]")
  status                      String                       @default("ACTIVE")
  createdAt                   DateTime                     @default(now())
  updatedAt                   DateTime                     @updatedAt
  deviceGroupIntervals        PunchRuleDeviceGroupInterval[]
}

model DeviceGroup {
  id                          Int                          @id @default(autoincrement())
  code                        String                       @unique
  name                        String
  description                 String?
  status                      String                       @default("ACTIVE")
  createdAt                   DateTime                     @default(now())
  updatedAt                   DateTime                     @updatedAt
  devices                     PunchDevice[]
  punchRuleIntervals          PunchRuleDeviceGroupInterval[]
}
```

**优点：**
- 数据结构清晰，易于理解和维护
- 支持数据库级别的唯一性约束
- 便于查询和索引
- 数据完整性由数据库保证

**缺点：**
- 需要修改数据库结构

**推荐方案：方案二（创建关联表）**

### 3.5 业务逻辑

#### 3.5.1 间隔匹配逻辑
当员工打卡时，系统按照以下逻辑匹配打卡间隔：

1. **确定打卡设备所属设备组**
   - 根据打卡记录的 `deviceId` 查询设备信息
   - 获取设备的 `groupId`

2. **查找匹配的打卡规则**
   - 按规则的优先级从高到低排序
   - 检查规则的条件（conditions）是否匹配
   - 找到第一个匹配的打卡规则

3. **确定使用的打卡间隔**
   - 在匹配的打卡规则中，查找该设备组是否有特定间隔配置
   - 如果有，使用设备组特定的间隔
   - 如果没有，使用打卡规则的全局间隔（`beforeShiftMins` 和 `afterShiftMins`）

4. **计算有效打卡时间**
   - 班次开始时间 - beforeShiftMins = 最早打卡时间
   - 班次结束时间 + afterShiftMins = 最晚打卡时间

#### 3.5.2 数据验证逻辑

**创建/更新打卡规则时：**
1. 验证设备组是否存在
2. 验证同一设备组不能重复配置（数据库唯一性约束）
3. 验证间隔时间为非负整数
4. 验证设备组状态为启用

**删除设备组时：**
1. 检查是否有打卡规则引用该设备组
2. 如果有，可以选择：
   - 级联删除相关配置
   - 禁用相关配置
   - 提示用户先处理引用关系

#### 3.5.3 数据一致性保证

1. **数据库层面**
   - 使用 `@@unique([punchRuleId, deviceGroupId])` 确保唯一性
   - 使用外键约束确保引用完整性
   - 使用 `onDelete: Cascade` 确保删除打卡规则时级联删除设备组间隔配置

2. **应用层面**
   - 在创建/更新前进行重复性检查
   - 在设备组状态变更时，通知相关模块
   - 定期清理无效的设备组配置

## 五、非功能性需求

### 5.1 性能要求
- 设备组间隔查询响应时间 < 100ms
- 打卡规则创建/更新响应时间 < 500ms
- 支持至少 1000 个打卡规则
- 支持每个打卡规则配置至少 50 个设备组间隔

### 5.2 安全要求
- 设备组间隔配置需要相应的权限控制
- 记录配置变更的操作日志
- 防止越权访问和修改

### 5.3 可用性要求
- 页面操作简单直观
- 提供清晰的错误提示
- 支持批量导入/导出功能（可选）

## 六、风险评估

### 6.1 技术风险
- **风险**：数据库迁移可能导致数据丢失
- **应对**：充分测试迁移脚本，做好数据备份

### 6.2 业务风险
- **风险**：现有打卡规则可能需要重新配置
- **应对**：保持向后兼容，全局配置仍然有效

### 6.3 性能风险
- **风险**：大量设备组间隔配置可能影响性能
- **应对**：优化查询逻辑，添加合适的索引
