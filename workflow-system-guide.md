# 可配置工作流系统使用指南

## 概述

本系统提供了一个完整的可配置工作流引擎，支援申请和报工申请可以通过可视化流程设计器配置审批流程，支持多种审批人策略、多人审批模式和条件分支。

## 核心功能

### 1. 可视化流程设计器

**访问路径**：工作台 → 流程定义 → 设计按钮

**功能特性**：
- 拖拽式节点设计
- 支持多种节点类型：开始、审批、条件、抄送、结束
- 可视化连线配置
- 实时预览

**节点类型**：

| 节点类型 | 说明 | 配置项 |
|---------|------|--------|
| 开始 | 流程的起始点 | 无 |
| 审批 | 需要人员审批的节点 | 审批方式、审批人 |
| 条件 | 根据条件分支流转 | 条件表达式 |
| 抄送 | 知会给相关人员 | 抄送人 |
| 结束 | 流程的终点 | 无 |

### 2. 审批人策略

系统支持 **9 种审批人获取策略**：

#### 2.1 指定用户 (SPECIFIC_USER)
直接指定具体的审批人。
```
配置示例：{ "userIds": [1, 2, 3] }
```

#### 2.2 角色 (ROLE)
根据角色获取审批人。
```
配置示例：{ "roleId": 5 }
说明：获取角色ID为5的所有用户
```

#### 2.3 部门负责人 (DEPT_LEADER)
获取指定部门的负责人。
```
配置示例：{ "deptId": 10 }
说明：获取部门ID为10的负责人
```

#### 2.4 发起人的直属领导 (INITIATOR_LEADER)
自动获取发起人所在部门的负责人。
```
配置示例：无需额外配置
说明：系统自动根据发起人的组织获取负责人
```

#### 2.5 上级领导 (SUPERIOR)
获取发起人的上级领导。
```
配置示例：{ "level": 1 }
说明：
- level=1: 直属上级
- level=2: 上上级
- 以此类推
```

#### 2.6 发起人自己 (SELF)
审批人是发起人本人（适用于确认环节）。
```
配置示例：无需额外配置
```

#### 2.7 表单字段 (FORM_FIELD)
从表单字段中获取审批人。
```
配置示例：{ "fieldName": "approverId" }
说明：从表单的 approverId 字段获取审批人ID
```

#### 2.8 多个角色 (MULTIPLE_ROLES)
获取多个角色的并集用户。
```
配置示例：{ "roleIds": [5, 6, 7] }
说明：获取角色5、6、7的所有用户（去重）
```

#### 2.9 部门+角色组合 (DEPT_AND_ROLE)
获取指定部门下拥有指定角色的用户。
```
配置示例：{ "deptId": 10, "roleId": 5 }
说明：获取部门10下拥有角色5的用户
```

### 3. 多人审批模式

系统支持 **3 种多人审批模式**：

#### 3.1 串行审批 (sequential)
审批人按顺序依次审批，必须按顺序。
```
场景：科长 → 处长 → 局长
特点：
- 必须按顺序审批
- 前一个人审批后，下一个人才能审批
- 所有人都审批通过才流转到下一节点
```

#### 3.2 并行审批 (parallel)
所有审批人都可以同时审批，必须所有人都审批通过。
```
场景：财务经理、法务经理、技术经理同时审批
特点：
- 所有人都可以同时审批
- 必须所有人都审批通过
- 任意一人驳回则流程结束
```

#### 3.3 或签 (or-sign)
任意一人审批通过即可。
```
场景：多个部门领导中任意一人审批
特点：
- 所有人都可以审批
- 第一人审批通过即流转
- 适用于替代审批场景
```

### 4. 条件分支

支持根据表单数据进行条件判断，实现不同的审批路径。

**条件配置示例**：
```javascript
{
  "field": "amount",          // 字段名
  "operator": "gt",           // 操作符
  "value": 10000              // 比较值
}
```

**支持的操作符**：

| 操作符 | 说明 | 示例 |
|-------|------|------|
| eq | 等于 | amount 等于 10000 |
| ne | 不等于 | type 不等于 "特殊" |
| gt | 大于 | amount 大于 10000 |
| gte | 大于等于 | amount 大于等于 10000 |
| lt | 小于 | amount 小于 10000 |
| lte | 小于等于 | amount 小于等于 10000 |
| in | 包含于 | type 在 ["A", "B", "C"] 中 |
| contains | 字符串包含 | comment 包含 "紧急" |

**条件分支场景示例**：
```
开始 → 金额判断
       ├─ < 1万 → 部门主管审批 → 结束
       ├─ 1万-10万 → 部门主管+财务审批 → 结束
       └─ > 10万 → 部门主管+财务+总经理审批 → 结束
```

## 使用流程

### 步骤1：创建流程定义

1. 进入"流程定义"页面
2. 点击"新增"按钮
3. 填写基本信息：
   - 流程编码：系统自动生成
   - 流程名称：如"支援申请审批流程"
   - 流程分类：选择"支援申请"或"报工申请"
   - 描述：流程说明

### 步骤2：设计流程

1. 在流程定义列表中，点击"设计"按钮
2. 进入流程设计器
3. 从工具栏拖拽节点到画布：
   - 点击"开始"按钮添加开始节点
   - 点击"审批"按钮添加审批节点
   - 点击"条件"按钮添加条件节点
   - 点击"抄送"按钮添加抄送节点
   - 点击"结束"按钮添加结束节点
4. 连接节点：
   - 从一个节点的连接点拖拽到另一个节点
   - 系统会自动创建连线
5. 配置节点：
   - 点击节点打开配置面板
   - 设置节点名称
   - 配置审批人/条件
   - 保存配置
6. 预览和保存：
   - 点击"预览"查看流程图
   - 点击"保存"保存流程设计

### 步骤3：发布流程

1. 返回流程定义列表
2. 找到设计好的流程（状态为"草稿"）
3. 点击"发布"按钮
4. 流程状态变为"已发布"

### 步骤4：使用流程

1. 在支援申请或报工申请创建页面
2. 选择已发布的工作流
3. 填写表单并提交
4. 系统自动按照设计的流程进行审批

## API接口

### 流程设计器

**保存流程设计**
```
POST /workflow/design
Request Body:
{
  "definitionId": 1,              // 流程定义ID（更新时传，新增时不传）
  "name": "流程名称",
  "category": "SUPPORT_REQUEST",   // 流程分类
  "description": "流程描述",
  "flowConfig": {
    "nodes": [...],
    "edges": [...]
  }
}
```

**获取流程设计**
```
GET /workflow/design/{id}
Response:
{
  "id": 1,
  "code": "SUPPORT_REQUEST_123",
  "name": "流程名称",
  "category": "SUPPORT_REQUEST",
  "flowConfig": {
    "nodes": [...],
    "edges": [...]
  },
  "nodes": [...],
  "edges": [...]
}
```

### 流程实例

**启动流程**
```
POST /workflow/instances/start
Request Body:
{
  "workflowCode": "SUPPORT_REQUEST_123",
  "title": "申请标题",
  "formData": { ... }
}
```

**审批操作**
```
POST /workflow/instances/approval
Request Body:
{
  "instanceId": 1,
  "action": "APPROVED",    // APPROVED, REJECTED, WITHDRAWN
  "comment": "审批意见"
}
```

## 常见场景配置

### 场景1：简单审批（直属领导审批）

```
开始 → 直属领导审批（发起人的直属领导）→ 结束
```

**配置**：
- 节点类型：审批
- 审批方式：串行审批
- 审批人策略：INITIATOR_LEADER

### 场景2：多级审批（科长→处长→局长）

```
开始 → 科长审批 → 处长审批 → 局长审批 → 结束
```

**配置**：
- 每个节点都是审批节点
- 审批方式：串行审批
- 审批人策略：
  - 科长：DEPT_LEADER (发起人所在科室)
  - 处长：SUPERIOR (level=1)
  - 局长：SUPERIOR (level=2)

### 场景3：会签审批（财务+法务+技术）

```
开始 → 三部门并行审批 → 结束
```

**配置**：
- 节点类型：审批
- 审批方式：并行审批
- 审批人策略：
  - 策略1：ROLE (roleId=财务角色)
  - 策略2：ROLE (roleId=法务角色)
  - 策略3：ROLE (roleId=技术角色)

### 场景4：条件分支（金额判断）

```
开始 → 金额判断
       ├─ < 1万 → 部门主管审批 → 结束
       ├─ 1-10万 → 部门主管+财务审批 → 结束
       └─ > 10万 → 部门主管+财务+总经理审批 → 结束
```

**配置**：
- 条件节点：根据 formData.amount 判断
- 不同分支连接到不同的审批节点

## 数据库表结构

### WorkflowDefinition（流程定义表）
- 存储流程的基本信息和流程图配置
- 状态：DRAFT（草稿）、PUBLISHED（已发布）、ARCHIVED（已归档）

### WorkflowNode（流程节点表）
- 存储流程中的节点信息
- 包含审批人策略、条件配置等

### WorkflowEdge（流程连线表）
- 存储节点之间的连接关系
- 支持条件配置

### WorkflowInstance（流程实例表）
- 存储流程运行时的实例数据
- 记录当前节点、状态等

### WorkflowApproval（审批记录表）
- 存储所有审批操作记录
- 记录审批人、审批动作、审批意见等

## 注意事项

1. **流程设计规范**：
   - 每个流程必须有且仅有一个开始节点
   - 每个流程必须有且仅有一个结束节点
   - 所有节点必须连通，不能有孤立的节点

2. **审批人配置**：
   - 每个审批节点必须配置至少一个审批人策略
   - 确保审批人配置有效，避免流程卡住

3. **条件分支**：
   - 条件表达式必须正确
   - 确保每个分支都有出路
   - 建议添加默认分支处理未匹配的情况

4. **流程修改**：
   - 已发布的流程不能直接修改
   - 需要修改时，建议创建新版本
   - 正在运行的流程实例不受影响

5. **性能优化**：
   - 避免创建过于复杂的流程
   - 合理设置审批节点数量
   - 定期清理已归档的流程实例

## 常见问题

**Q1：流程发布后还能修改吗？**
A：已发布的流程不能直接修改。如需修改，建议创建新的流程定义或版本管理功能。

**Q2：如何处理流程卡住的情况？**
A：可以通过流程实例管理页面，强制终止或重新激活流程。

**Q3：审批人离职了怎么办？**
A：系统会自动根据审批人策略重新计算审批人，确保流程继续流转。

**Q4：如何查看流程的审批进度？**
A：在流程实例详情页面可以查看完整的审批记录和当前状态。

**Q5：支持流程撤回吗？**
A：支持。只有发起人可以撤回，且必须是在没有人审批之前。

4. **移动端**：提供移动端审批界面
5. **数据分析**：流程效率分析、审批统计
6. **脚本引擎**：支持自定义脚本计算审批人
7. **子流程**：支持在一个流程中调用另一个流程
8. **回退机制**：支持审批回退到上一节点
