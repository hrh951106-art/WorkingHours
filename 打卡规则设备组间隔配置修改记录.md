# 打卡规则设备组间隔配置修改记录

## 修改日期
2026-04-08

## 修改概述
本次修改为打卡规则新增了设备组刷卡间隔配置功能，支持针对不同设备组设置独立的打卡间隔时间，提高了打卡规则的灵活性和可配置性。

## 需求背景
1. 打卡规则（PunchRule）仅支持全局的打卡间隔配置
2. 无法针对特定设备组进行差异化配置
3. 需要支持为不同设备组设置独立的班前班后间隔时间

## 功能需求
1. 支持针对设备组配置刷卡间隔
2. 支持配置多套间隔（一个打卡规则可配置多个设备组）
3. 设备组非必选
4. 刷卡间隔非必选
5. 同一打卡规则中一个设备组不能配置两个刷卡间隔（唯一性约束）

## 修改内容

### 1. 数据库修改

#### 1.1 创建新表
在 `backend/prisma/schema.prisma` 中新增了 `PunchRuleDeviceGroupInterval` 模型：

```prisma
model PunchRuleDeviceGroupInterval {
  id               Int         @id @default(autoincrement())
  punchRuleId      Int
  deviceGroupId    Int
  deviceGroupCode  String
  deviceGroupName  String
  beforeShiftMins  Int         @default(0)
  afterShiftMins   Int         @default(0)
  status           String      @default("ACTIVE")
  createdAt        DateTime    @default(now())
  updatedAt        DateTime    @updatedAt
  punchRule        PunchRule   @relation(fields: [punchRuleId], references: [id], onDelete: Cascade)
  deviceGroup      DeviceGroup @relation(fields: [deviceGroupId], references: [id])

  @@unique([punchRuleId, deviceGroupId])
  @@index([punchRuleId])
  @@index([deviceGroupId])
}
```

**特性说明：**
- `punchRuleId` 和 `deviceGroupId` 组成唯一性约束，确保同一打卡规则中一个设备组只能配置一次
- 使用外键关联 `PunchRule` 和 `DeviceGroup` 表
- 支持级联删除（删除打卡规则时自动删除相关设备组间隔配置）

#### 1.2 修改现有模型

**PunchRule 模型：**
```prisma
model PunchRule {
  // ...原有字段...
  deviceGroupIntervals PunchRuleDeviceGroupInterval[]
}
```

**DeviceGroup 模型：**
```prisma
model DeviceGroup {
  // ...原有字段...
  punchRuleIntervals PunchRuleDeviceGroupInterval[]
}
```

#### 1.3 数据库迁移
使用 `npx prisma db push` 将 schema 变更应用到数据库。

### 2. 后端修改

#### 2.1 Service层修改
文件：`backend/src/modules/calculate/calculate.service.ts`

**修改的方法：**

1. **getPunchRules()** - 查询打卡规则时包含设备组间隔配置
```typescript
async getPunchRules() {
  return this.prisma.punchRule.findMany({
    where: { status: 'ACTIVE' },
    include: {
      deviceGroupIntervals: true,
    },
    orderBy: { priority: 'asc' },
  });
}
```

2. **createPunchRule()** - 创建打卡规则时支持设备组间隔配置
- 添加了设备组间隔配置的验证（检查同一设备组是否重复配置）
- 创建打卡规则后，循环创建设备组间隔配置记录
- 自动获取设备组的 code 和 name

3. **updatePunchRule()** - 更新打卡规则时支持设备组间隔配置
- 添加了设备组间隔配置的验证
- 更新时先删除原有的设备组间隔配置，再创建新的配置

4. **新增 getPunchRuleById()** - 根据ID查询打卡规则及其设备组间隔配置
```typescript
async getPunchRuleById(id: number) {
  return this.prisma.punchRule.findUnique({
    where: { id },
    include: {
      deviceGroupIntervals: true,
    },
  });
}
```

**业务逻辑说明：**
- 在创建/更新打卡规则时，会检查 `deviceGroupIntervals` 数组中是否存在重复的 `deviceGroupId`
- 如果存在重复，抛出错误："同一打卡规则中，一个设备组不能配置多个刷卡间隔"
- 对每个设备组间隔配置，会验证设备组是否存在
- 设备组间隔配置的 `beforeShiftMins` 和 `afterShiftMins` 默认值为 0

#### 2.2 Controller层
文件：`backend/src/modules/calculate/calculate.controller.ts`

Controller层无需修改，现有接口已支持新的数据结构。

## API接口说明

### 创建打卡规则
**POST /api/calculate/punch-rules**

请求体示例：
```json
{
  "code": "RULE_001",
  "name": "生产车间打卡规则",
  "priority": 1,
  "conditions": "{}",
  "beforeShiftMins": 30,
  "afterShiftMins": 30,
  "status": "ACTIVE",
  "deviceGroupIntervals": [
    {
      "deviceGroupId": 1,
      "beforeShiftMins": 60,
      "afterShiftMins": 60
    },
    {
      "deviceGroupId": 2,
      "beforeShiftMins": 90,
      "afterShiftMins": 90
    }
  ]
}
```

响应示例：
```json
{
  "id": 1,
  "code": "RULE_001",
  "name": "生产车间打卡规则",
  "priority": 1,
  "conditions": "{}",
  "beforeShiftMins": 30,
  "afterShiftMins": 30,
  "status": "ACTIVE",
  "deviceGroupIntervals": [
    {
      "id": 1,
      "punchRuleId": 1,
      "deviceGroupId": 1,
      "deviceGroupCode": "GROUP_A",
      "deviceGroupName": "车间A设备组",
      "beforeShiftMins": 60,
      "afterShiftMins": 60,
      "status": "ACTIVE"
    }
  ]
}
```

### 更新打卡规则
**PUT /api/calculate/punch-rules/:id**

请求体示例：
```json
{
  "name": "生产车间打卡规则（已修改）",
  "deviceGroupIntervals": [
    {
      "deviceGroupId": 1,
      "beforeShiftMins": 60,
      "afterShiftMins": 60
    },
    {
      "deviceGroupId": 3,
      "beforeShiftMins": 120,
      "afterShiftMins": 120
    }
  ]
}
```

### 获取打卡规则列表
**GET /api/calculate/punch-rules**

响应包含 `deviceGroupIntervals` 数组。

## 注意事项

1. **数据完整性**
   - 删除设备组时，需要考虑关联的打卡规则间隔配置
   - 建议在删除设备组前检查是否有打卡规则引用

2. **性能考虑**
   - 当设备组间隔配置较多时，查询性能可能受影响
   - 已添加索引优化查询性能

3. **向后兼容**
   - 全局间隔配置（`beforeShiftMins` 和 `afterShiftMins`）仍然有效
   - 未配置设备组特定间隔时，使用全局间隔
