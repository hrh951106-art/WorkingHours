# 基于优先级的成对收卡逻辑测试指南

## 概述

本文档用于验证基于优先级的成对收卡逻辑的正确性，包括4种标准情况和各种边界情况。

## 业务规则

### 成对收卡优先级

```
优先级1：子劳动力账户 + 进出标记 都相同
    ↓
优先级2：子劳动力账户 相同
    ↓
优先级3：进出标记 相同
    ↓
优先级4：无标记（第一笔视为进）
```

### 4种标准情况

| 情况 | 子劳动力账户 | 进出标记 | 成对依据 | 存储账户 |
|------|-------------|----------|----------|----------|
| 1 | ✅有 | ✅有 | 人员 + 标记 + 账户 | 带账户 |
| 2 | ❌无 | ✅有 | 人员 + 标记 | 不带账户 |
| 3 | ✅有 | ❌无 | 人员 + 账户 | 带账户 |
| 4 | ❌无 | ❌无 | 人员（相邻） | 不带账户 |

## 测试场景

### 场景1：标准情况1 - 有子劳动力账户 + 有进出标记

**输入数据**：
```

09:05:00 - 出 (账户B, 设备2)
```

**预期结果**：
```
优先级1：
- 对1：账户A, 08:00:00 (进) + 09:00:00 (出)
```

**输入数据**：
```

08:05:00 - 进 (账户A)  # 重复进
08:10:00 - 进 (账户B)
09:00:00 - 出 (账户A)
09:05:00 - 出 (账户B)
09:10:00 - 出 (账户C)
```

**预期结果**：
```
优先级1：
- 对1：账户A, 08:00:00 (进) + 09:00:00 (出)
- 对2：账户B, 08:10:00 (进) + 09:05:00 (出)

单卡：
- 单卡1：账户A, 08:05:00 (进, 已过滤或未配对)
- 单卡2：账户C, 09:10:00 (出)
```

**注意**：如果启用了摆卡间隔过滤（如5分钟），08:05:00的进卡会被过滤掉。

### 验证成对结果

```sql
-- 检查每个摆卡记录的进出标记和账户
SELECT
  pp.id,
  pp.employeeNo,
  pp.pairDate,
  pp.inPunchTime,
  pp.outPunchTime,
  pp.workHours,
  pp.accountId,
  in_pr.punchType as inPunchType,
  out_pr.punchType as outPunchType,
  in_pr.accountId as inPunchAccountId,
  out_pr.accountId as outPunchAccountId
FROM PunchPair pp
LEFT JOIN PunchRecord in_pr ON pp.inPunchId = in_pr.id
LEFT JOIN PunchRecord out_pr ON pp.outPunchId = out_pr.id
WHERE pp.employeeNo = 'TEST001'
  AND pp.pairDate = '2025-01-15'
ORDER BY pp.inPunchTime;
```

## 注意事项

### 1. 账户ID的处理

- 有子劳动力账户：摆卡记录的 accountId 为对应账户ID
- 无子劳动力账户：摆卡记录的 accountId 为 null
- 优先级1和2的配对会带账户ID
- 优先级3和4的配对不带账户ID

### 2. 进出标记的处理

- 有标记：使用打卡记录的原始 punchType
- 无标记：自动识别，第一张为进，第二张为出，依此类推
- 优先级1和3的配对依赖进出标记
- 优先级2和4的配对不依赖进出标记

### 3. 摆卡间隔的影响

- 如果配置了摆卡间隔（如5分钟）：
  - 在成对收卡前会进行过滤
  - 在成对收卡时会检查最小时间间隔
- 如果摆卡间隔为0：
  - 不过滤打卡记录
  - 直接按顺序配对

### 4. 单卡的处理

- 只有签入：创建单卡摆卡，outPunchId 为 null
- 只有签退：创建单卡摆卡，inPunchId 为 null
- 单卡也会触发工时计算（工时为0）

### 5. 工时计算

- 完整配对：计算进出时间差
- 单卡：工时为0
- 工时单位：小时（保留两位小数）

## 已知问题

### 1. 性能影响

- 基于优先级的成对收卡比简单配对更耗时
- 多次遍历打卡记录进行不同优先级的匹配
- 建议：优化算法，减少遍历次数

### 2. 边界情况

- 奇数个打卡记录的处理（最后一个单独处理）
- 所有打卡记录都无法配对的情况
- 建议：增加更多测试场景

### 3. 账户合并的影响

- 如果打卡记录的账户被合并，成对收卡会使用合并后的账户ID
- 需要确保账户合并在成对收卡之前完成
- 建议：验证账户合并和成对收卡的集成
