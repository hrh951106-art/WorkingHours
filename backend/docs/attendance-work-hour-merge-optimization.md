# 考勤工时计算优化 - 相邻账户合并

## 📋 优化背景

之前的考勤工时计算逻辑会为每个班段生成独立的工时记录，导致相同劳动力账户且时间相邻的工时被拆分成多条记录。

例如：
- 第一段 08:00~09:00 账户A
- 第二段 09:00~12:00 账户A

应该合并为：08:00~12:00 账户A

## 🔧 优��内容

### 修改文件

`/backend/src/modules/calculate/attendance-work-hour.service.ts`

### 核心逻辑

在 `calculateDaily` 方法中添加了工时合并逻辑：

1. **收集所有工时结果**：为每个打卡收卡结果计算工时
2. **合并相邻工时**：调用 `mergeAdjacentWorkHours` 方法合并相同账户且时间相邻的工时
3. **保存合并后的结果**：保存到数据库

### 合并规则

`mergeAdjacentWorkHours` 方法实现以下合并规则：

1. **相同账户**：`accountName` 必须相同
2. **时间相邻**：当前记录的开始时间 = 前一条记录的结束时间
3. **合并方式**：
   - `startTime` = 第一条记录的开始时间
   - `endTime` = 最后一条记录的结束时间
   - `workHours` = 所有记录的工时之和
   - `amount` = 保存时重新计算（不直接相加）

### 代码实现

```typescript
// 在 calculateDaily 方法中（第52-54行）
// 2.5. ✅ 合并相同劳动力账户且时间相邻的工时结果
const mergedResults = this.mergeAdjacentWorkHours(workHourResults);
if (workHourResults.length !== mergedResults.length) {
  this.logger.log(`合并工时结果: ${workHourResults.length} 条 -> ${mergedResults.length} 条`);
}

// 3. 保存或更新工时结果
const savedResults = await this.saveWorkHourResults(mergedResults, batchId);
```

### 新增方法

1. **`mergeAdjacentWorkHours(workHourResults: any[]): any[]`**
   - 合并相同账户且时间相邻的工时结果
   - 返回合并后的工时结果数组

2. **`mergeWorkHourGroup(group: any[]): any`**
   - 将一组可合并的工时结果合并为一条
   - 计算总工时，调整开始和结束时间

## 🔍 关键点

1. **金额不直接合并**：由于金额是根据工时、出勤代码和账户路径动态计算的，合并时不直接相加金额，而是在保存时重新计算

2. **时间精确匹配**：判断时间相邻时，使用毫秒级时间戳比较（`currentTime === previousEndTime`），确保精确匹配

3. **保留原始信息**：合并时保留第一条记录的基本信息（员工、日期、班次等），只调整时间和工时

4. **日志记录**：记录合并前后的记录数，方便调试和监控

## ✅ 优势

1. **减少数据冗余**：相同账户且时间相邻的工时合并为一条，减少数据库存储
2. **提高查询效率**：减少记录数量，提高查询和显示速度
3. **便于统计分析**：合并后的工时记录更符合业务逻辑，便于统计和报表
4. **保持灵活性**：不同账户或时间不相邻的工时仍然独立保存

## 📝 注意事项

- 只合并**相同账户**且**时间相邻**的工时
- 不同账户的工时不会合并（即使时间相邻）
- 时间不相邻的同账户工时不会合并
- 合并信息会记录在 `remark` 字段中，便于追溯
