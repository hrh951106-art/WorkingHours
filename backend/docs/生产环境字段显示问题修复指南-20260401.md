# 生产环境字段显示问题修复指南

> **问题时间**: 2026-04-01
> **问题现象**: 新增人员时性别等字段未显示
> **根本原因**: 生产环境fieldType=SELECT_SINGLE，前端只识别SYSTEM

---

## 🔍 问题分析

### 环境对比

| 项目 | 本地环境 | 生产环境 |
|-----|---------|---------|
| **数据库fieldType** | SYSTEM | SELECT_SINGLE |
| **前端过滤逻辑** | `fieldType === 'SYSTEM'` | `fieldType === 'SYSTEM'` |
| **CustomField记录** | 无 | 无/有 |
| **字段显示** | ✅ 显示 | ❌ 不显示 |

### 问题根源

```
生产环境数据状态：
├─ EmployeeInfoTabField.fieldType = 'SELECT_SINGLE' ❌
├─ CustomField 记录缺失或有
└─ DataSource 有数据

前端代码状态：
└─ 只过滤 fieldType === 'SYSTEM' 的字段

结果：SELECT_SINGLE字段被过滤掉 → 不显示
```

---

## 📋 排查步骤

### 步骤1：执行排查SQL

在生产数据库执行：

```bash
psql -U postgres -h localhost -d your_database -f backend/scripts/production-diagnose.sql
```

### 步骤2：分析诊断结果

#### 诊断1：检查字段类型分布

```sql
SELECT fieldCode, fieldName, fieldType
FROM "EmployeeInfoTabField"
WHERE fieldCode IN ('gender', 'nation', 'maritalStatus')
ORDER BY fieldCode;
```

**如果结果是 `fieldType = 'SELECT_SINGLE'`**：
- ✅ 确认问题：前端不识别此类型
- → 需要执行修复方案

**如果结果是 `fieldType = 'SYSTEM'`**：
- 问题可能在其他地方（如CustomField记录缺失）
- → 检查CustomField表

#### 诊断2：检查CustomField记录

```sql
SELECT code, name, type, "dataSourceId"
FROM "CustomField"
WHERE code IN ('gender', 'nation', 'maritalStatus');
```

**如果返回0行**：
- ❌ 缺少CustomField记录
- → 需要创建CustomField记录

**如果返回数据但dataSourceId为NULL**：
- ❌ 未关联数据源
- → 需要关联数据源

#### 诊断3：检查数据源

```sql
SELECT ds.code, ds.name, COUNT(dso.id) as optionCount
FROM "DataSource" ds
LEFT JOIN "DataSourceOption" dso ON dso."dataSourceId" = ds.id
WHERE ds.code IN ('gender', 'nation')
GROUP BY ds.code, ds.name;
```

**期望结果**：每个数据源都有选项（optionCount > 0）

## 🔧 修复方案

### 方案A：快速修复（紧急恢复）

**适用场景**：生产故障需要紧急恢复

**操作步骤**：

1. **备份数据库**
   ```bash
   pg_dump -U postgres -h localhost -d your_database > backup_$(date +%Y%m%d).sql
   ```

2. **将字段类型改回SYSTEM**
   ```sql
   BEGIN;

   UPDATE "EmployeeInfoTabField"
   SET "fieldType" = 'SYSTEM'
   WHERE "fieldType" = 'SELECT_SINGLE'
   AND "fieldCode" IN (
       'gender', 'nation', 'maritalStatus', 'politicalStatus',
       'educationLevel', 'educationType', 'jobLevel',
       'employeeType', 'resignationReason', 'familyRelation', 'A05'
   );

   COMMIT;
   ```

3. **验证结果**
   ```sql
   SELECT fieldCode, fieldName, fieldType
   FROM "EmployeeInfoTabField"
   WHERE fieldCode IN ('gender', 'nation', 'maritalStatus')
   ORDER BY fieldCode;
   -- 应该返回 fieldType = 'SYSTEM'
   ```

4. **清除浏览器缓存并刷新页面**
   - Chrome: Ctrl+Shift+Delete
   - 验证字段显示

**优点**：
- ✅ 快速恢复显示
- ✅ 不需要更新前端代码
- ✅ 风险小

**缺点**：
- ❌ 无法使用数据源功能
- ❌ 下拉框变输入框
- ❌ 需要后续完整修复

### 方案B：完整修复（推荐）

**适用场景**：标准更新流程

**前置条件**：
- ✅ 已备份数据库
- ✅ 已更新前端代码（支持SELECT_SINGLE）
- ✅ 已重新构建前端

**操作步骤**：

1. **备份数据库**
   ```bash
   pg_dump -U postgres -h localhost -d your_database > backup_$(date +%Y%m%d).sql
   ```

2. **执行完整修复SQL**
   ```bash
   psql -U postgres -h localhost -d your_database -f backend/scripts/production-fix-fieldType.sql
   ```

   此SQL会：
   - ✅ 为缺失字段创建CustomField记录
   - ✅ 统一字段类型为SELECT_SINGLE
   - ✅ 关联数据源

3. **验证修复结果**
   ```sql
   -- 验证1：检查CustomField记录
   SELECT COUNT(*) FROM "CustomField"
   WHERE code IN (
       'gender', 'nation', 'maritalStatus', 'politicalStatus',
       'educationLevel', 'educationType', 'jobLevel',
       'employeeType', 'resignationReason', 'familyRelation', 'A05'
   );
   -- 应该返回 11

   -- 验证2：检查字段类型
   SELECT fieldCode, fieldType
   FROM "EmployeeInfoTabField"
   WHERE fieldCode IN ('gender', 'nation', 'maritalStatus')
   ORDER BY fieldCode;
   -- 应该返回 fieldType = 'SELECT_SINGLE'

   -- 验证3：检查数据源关联
   SELECT
       cf.code,
       cf.name,
       ds.code AS dataSourceCode,
       COUNT(dso.id) AS optionCount
   FROM "CustomField" cf
   LEFT JOIN "DataSource" ds ON ds.id = cf."dataSourceId"
   LEFT JOIN "DataSourceOption" dso ON dso."dataSourceId" = ds.id
   WHERE cf.code IN ('gender', 'nation')
   GROUP BY cf.code, cf.name, ds.code;
   -- optionCount应该 > 0
   ```

4. **部署前端代码**
   ```bash
   # 停止服务
   systemctl stop nginx

   # 部署新构建的前端
   cp -r frontend/dist/* /path/to/nginx/html/

   # 启动服务
   systemctl start nginx
   ```

5. **清除浏览器缓存并验证**
   - Chrome: Ctrl+Shift+Delete
   - 打开新增人员页面
   - 验证性别、民族等字段显示为下拉框
   - 验证下拉选项正确加载
   - 测试保存功能

**优点**：
- ✅ 完整功能，支持数据源
- ✅ 下拉框正常工作
- ✅ 长期稳定

**缺点**：
- ❌ 需要更新前端代码
- ❌ 执行步骤较多

## 📞 问题处理

### Q1: 执行SQL后字段仍不显示

**排查**:
1. 检查前端是否已更新并重新部署
2. 清除浏览器缓存（Ctrl+Shift+Delete）
3. 检查前端控制台是否有错误
4. 验证API返回的fieldType值

```bash
# 测试API
curl -H "Authorization: Bearer YOUR_TOKEN" \
  http://your-backend/api/hr/employee-info-tabs/for-display
```

### Q2: 字段显示但无下拉选项

**排查**:
1. 检查CustomField.dataSourceId是否正确
2. 检查DataSource表是否有记录
3. 检查DataSourceOption表是否有选项

```sql
-- 检查数据源配置
SELECT
    cf.code,
    cf."dataSourceId",
    ds.code AS dataSourceCode,
    COUNT(dso.id) AS optionCount
FROM "CustomField" cf
LEFT JOIN "DataSource" ds ON ds.id = cf."dataSourceId"
LEFT JOIN "DataSourceOption" dso ON dso."dataSourceId" = ds.id
WHERE cf.code = 'gender'
GROUP BY cf.code, cf."dataSourceId", ds.code;
```

### Q3: 保存时提示"性别值无效"

**原因**: 后端代码未更新

**解决**:
1. 确认后端代码已更新（去掉gender硬编码验证）
2. 重新构建并部署后端
3. 重启后端服务
