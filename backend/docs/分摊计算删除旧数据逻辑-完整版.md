# 分摊计算删除旧数据逻辑（完整版）

## 修复时间
2026-03-31（第二次修复）

## 问题描述
在执行分摊计算时，不仅需要删除分摊结果记录（AllocationResult），还需要删除间接工时的工时记录（CalcResult），因为这些记录会在工时明细管理页面显示。

### 问题场景
1. **第一次分摊**：
   - 产生 10 条 AllocationResult（分摊关系记录）
   - 产生 10 条 CalcResult（间接工时记录）
   - 在工时明细管理页面可以看到这 10 条间接工时记录

2. **修改配置后第二次分摊**：
   - 如果只删除 AllocationResult，不删除 CalcResult
   - 工时明细管理页面仍会显示旧的 10 条间接工时记录
   - 同时又产生新的 15 条间接工时记录
   - **结果：页面显示 25 条记录，但实际应该只有 15 条**

## 修复方案

### 完整的删除逻辑

```javascript
// ========== 清理旧数据：删除同一配置的旧分摊结果和间接工时记录 ==========
const uniqueDates = Object.keys(groupedByDate);
const dateObjects = uniqueDates.map(dateKey => new Date(dateKey));

console.log(`[分摊计算] 开始清理旧分摊数据...`);
console.log(`  - 配置ID: ${config.id}`);
console.log(`  - 日期范围: ${uniqueDates.join(', ')}`);
console.log(`  - 日期数量: ${uniqueDates.length}`);

// 1. 删除旧的分摊结果记录（AllocationResult）
const deletedAllocationResults = await this.prisma.allocationResult.deleteMany({
  where: {
    configId: config.id,
    recordDate: { in: dateObjects },
  },
});

console.log(`[分摊计算] 清理分摊结果: 删除 ${deletedAllocationResults.count} 条 AllocationResult 记录`);

// 2. 删除旧的间接工时记录（CalcResult）
// 这些记录的特征：
// - calcDate 在指定日期范围内
// - attendanceCodeId 是间接工时的出勤代码ID
// - status 是 'PENDING'（分摊产生的记录）
const deletedCalcResults = await this.prisma.calcResult.deleteMany({
  where: {
    calcDate: { in: dateObjects },
    attendanceCodeId: indirectHoursAttendanceCodeId,
    status: 'PENDING', // 只删除分摊产生的记录，不影响其他记录
  },
});

console.log(`[分摊计算] 清理间接工时记录: 删除 ${deletedCalcResults.count} 条 CalcResult 记录`);

const totalDeleted = deletedAllocationResults.count + deletedCalcResults.count;
console.log(`[分摊计算] 清理完成: 共删除 ${totalDeleted} 条旧记录（分摊结果 + 间接工时）`);

if (totalDeleted > 0) {
  console.log(`[分摊计算] 旧数据已清理，将生成新的分摊结果`);
} else {
  console.log(`[分摊计算] 没有找到旧分摊结果，这是首次分摊`);
}
```

### 删除范围说明

| 表名 | 删除条件 | 说明 |
|------|---------|------|
| **AllocationResult** | configId + recordDate | 分摊关系记录 |
| **CalcResult** | calcDate + attendanceCodeId + status | 间接工时的工时记录 |
| CalcResult（源数据） | ❌ 不删除 | 被分摊的原始工时记录，必须保留 |

### CalcResult 删除条件详解

```javascript
where: {
  calcDate: { in: dateObjects },                    // 指定日期范围
  attendanceCodeId: indirectHoursAttendanceCodeId, // 间接工时的出勤代码
  status: 'PENDING',                               // 只删除分摊产生的记录
}
```

**为什么要加 `status: 'PENDING'`？**
- 分摊产生的间接工时记录，status 默认为 'PENDING'
- 手动创建或审核通过的工时记录，status 不会是 'PENDING'
- 这样可以避免误删其他正常的工时记录

## 分摊计算产生的记录类型

### 1. AllocationResult（分摊关系记录）
记录分摊的映射关系：
- 源员工 → 目标对象（产线、车间等）
- 分摊依据、权重、比例
- 分配的工时数

### 2. CalcResult（间接工时记录）
记录分摊后的工时数据：
- 员工号、日期、班次
- 出勤代码（间接工时）
- 实际工时（分配的工时数）
- 账户信息（目标账户）
- 状态：PENDING

**创建位置**：
- `executeActualHoursAllocation`（第2037行）
- `executeActualYieldAllocation`（第2269行）
- `executeEquivalentYieldAllocation`（第2501行）
- `executeStandardHoursAllocation`（第2747行）

**示例记录**：
```javascript
{
  employeeNo: 'E001',
  calcDate: '2024-01-01',
  shiftId: 2,
  shiftName: '白班',
  attendanceCodeId: 5, // 间接工时的出勤代码ID
  standardHours: 0,
  actualHours: 2.5, // 分配的间接工时
  overtimeHours: 0,
  leaveHours: 0,
  absenceHours: 0,
  accountHours: '[{"accountId": 10, "accountName": "产线A", "hours": 2.5}]',
  accountId: 10, // 目标账户（产线A）
  accountName: '产线A',
  exceptions: '[]',
  status: 'PENDING', // 重要：标识这是分摊产生的记录
}
```

## 数据一致性保证

### 1. 完整删除
- ✅ 删除分摊关系记录（AllocationResult）
- ✅ 删除间接工时记录（CalcResult）

### 2. 精准删除
- ✅ 按配置ID + 日期范围删除 AllocationResult
- ✅ 按日期 + 出勤代码 + 状态删除 CalcResult

- ❌ 不删除被分摊的原始工时记录
- ❌ 不删除手动创建的工时记录

### 4. 批量删除
- 使用批量删除提高性能
- 一次性删除所有相关日期的数据
- 避免循环删除的性能问题

## 注意事项

1. **间接工时出勤代码**：
   - 必须在通用配置中正确设置间接工时的出勤代码
   - 删除逻辑依赖这个出勤代码ID

2. **状态标识**：
   - 分摊产生的 CalcResult 记录，status 必须是 'PENDING'
   - 避免误删其他状态的记录

3. **批量删除**：
   - 使用 `deleteMany` 而不是循环 `delete`
   - 提高删除性能

4. **删除顺序**：
   - 先删除 AllocationResult
   - 再删除 CalcResult
   - 顺序不重要（因为两者没有外键约束）
