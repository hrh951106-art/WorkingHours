# 基于优先级的成对收卡逻辑实现总结

## 完成时间
2025-01-15

## 实现概述

本次更新实现了基于优先级的成对收卡逻辑，支持4种标准情况，确保打卡记录能够按照业务规则正确配对。

## 核心功能

### 成对收卡优先级

```
优先级1：子劳动力账户 + 进出标记 都相同
    ↓
优先级2：子劳动力账户 相同
    ↓
优先级3：进出标记 相同
    ↓
优先级4：无标记（第一笔视为进）
```

### 4种标准情况

| 情况 | 子劳动力账户 | 进出标记 | 成对依据 | 存储账户 |
|------|-------------|----------|----------|----------|
| 1 | ✅有 | ✅有 | 人员 + 标记 + 账户 | 带账户 |
| 2 | ❌无 | ✅有 | 人员 + 标记 | 不带账户 |
| 3 | ✅有 | ❌无 | 人员 + 账户 | 带账户 |
| 4 | ❌无 | ❌无 | 人员（相邻） | 不带账户 |

## 业务流程

### 1. 优先级1：账户+标记相同

**适用情况**：有子劳动力账户 + 有进出标记

**流程**：
1. 按账户ID和进出标记分组
2. 在每个分组内进行配对
3. 检查时间间隔（如果配置了摆卡间隔）
4. 创建带账户ID的摆卡记录

**示例**：
```
账户A+进: [08:00, 08:05]
账户A+出: [09:00, 09:05]
配对结果：
- (账户A) 08:00(进) + 09:00(出)
- (账户A) 08:05(进) + 09:05(出)
```

### 2. 优先级2：账户相同

**适用情况**：有子劳动力账户 + 无进出标记

**流程**：
1. 按账户ID分组（不考虑进出标记）
2. 在每个账户组内进行配对
3. 自动识别第一张为进，第二张为出
4. 创建带账户ID的摆卡记录

**示例**：
```
账户A: [08:00, 09:00, 10:00]
配对结果：
- (账户A) 08:00(进) + 09:00(出)
- (账户A) 10:00(进) + [单卡]
```

### 3. 优先级3：标记相同

**适用情况**：无子劳动力账户 + 有进出标记

**流程**：
1. 所有打卡记录都没有账户ID
2. 按进出标记分组（进/出）
3. 在进/出组内进行配对
4. 创建不带账户ID的摆卡记录

**示例**：
```
进: [08:00, 08:05]
出: [09:00, 09:05]
配对结果：
- (无账户) 08:00(进) + 09:00(出)
- (无账户) 08:05(进) + 09:05(出)
```

### 4. 优先级4：相邻配对

**适用情况**：无子劳动力账户 + 无进出标记

**流程**：
1. 合并所有打卡记录
2. 按时间排序
3. 第一张视为进，第二张视为出，依此类推
4. 创建不带账户ID的摆卡记录

**示例**：
```
所有记录: [08:00, 08:05, 09:00, 09:05]
配对结果：
- (无账户) 08:00(进) + 08:05(出)
- (无账户) 09:00(进) + 09:05(出)
```

## 性能分析

### 时间复杂度

- 优先级1：O(n×m) - 账户和标记分组后的匹配
- 优先级2：O(n×m) - 账户分组后的匹配
- 优先级3：O(n×m) - 标记分组后的匹配
- 优先级4：O(n) - 相邻配对

总体：O(n×m)，其中n为签入数，m为签退数

### 空间复杂度

- 使用Set记录已使用的打卡记录：O(n+m)
- 分组Map：O(g)，g为分组数

总体：O(n+m+g)

### 优化建议

1. **提前退出**：如果所有打卡记录都已配对，提前退出
2. **索引优化**：为账户ID和进出标记创建索引
3. **批量处理**：批量创建摆卡记录，减少数据库操作
4. **缓存分组**：缓存分组结果，避免重复计算

## 已知问题

### 1. 性能影响

基于优先级的成对收卡比简单配对更耗时：
- 多次遍历打卡记录
- 多次分组和匹配操作
- 建议：优化算法，减少遍历次数

### 2. 复杂场景

某些复杂场景可能需要进一步优化：
- 跨天打卡的配对
- 连续多次同一类型打卡
- 异常时间间隔的处理

### 3. 日志输出

日志输出较多，可能影响性能：
- 建议：添加日志级别控制
- 建议：生产环境减少详细日志
