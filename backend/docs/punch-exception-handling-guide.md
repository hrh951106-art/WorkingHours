# 刷卡异常情况处理指南

## 概述

本文档详细说明了刷卡数据中的异常情况处理逻辑，包括有/无劳动力账户和有/无进出标记的各种组合情况。

## 异常情况分类

### 异常情况矩阵

| 序号 | 劳动力账户 | 进出标记 | 处理逻辑 | 优先级 |
|------|-----------|----------|----------|--------|
| 1 | ✅有 | ✅有 | 账户一致 + 进出标记成对 | 1 |
| 2 | ✅有 | ❌无 | 劳动力账户一致成对 | 2 |
| 3 | ❌无 | ✅有 | 进出标记成对 | 3 |
| 4 | ❌无 | ❌无 | 第一笔进，第二笔出（相邻配对） | 4 |

## 处理流程

### 总体流程

```
1. 分离签入和签退打卡记录
   ↓
2. 按优先级依次尝试配对
   ↓
3. 优先级1：账户+进出标记都相同
   ↓
4. 优先级2：账户相同（忽略进出标记）
   ↓
5. 优先级3：进出标记相同（忽略账户）
   ↓
6. 优先级4：相邻配对（时间顺序）
   ↓
7. 处理未配对的打卡记录（单卡）
```

### 详细处理逻辑

#### 优先级1：账户一致 + 进出标记成对

**触发条件**：
- 打卡记录有劳动力账户ID（accountId != null）
- 打卡记录有进出标记（punchType IN/OUT）

**处理步骤**：
1. 按 `accountId + punchType` 分组
2. 在每个分组内进行配对
3. 检查时间间隔（如果配置了摆卡间隔）
4. 创建带账户ID的摆卡记录

**示例**：
```
输入数据：
08:00:00 - 进 (账户A)
08:05:00 - 进 (账户B)
09:00:00 - 出 (账户A)
09:05:00 - 出 (账户B)

处理结果：
账户A+进: [08:00:00]
账户A+出: [09:00:00]
  → 配对1: 账户A, 08:00:00(进) + 09:00:00(出)

账户B+进: [08:05:00]
账户B+出: [09:05:00]
  → 配对2: 账户B, 08:05:00(进) + 09:05:00(出)
```

#### 优先级2：劳动力账户一致成对

**触发条件**：
- 打卡记录有劳动力账户ID（accountId != null）
- 打卡记录无进出标记（punchType 为空或UNKNOWN）
- 或者优先级1处理后的剩余记录

**处理步骤**：
1. 按 `accountId` 分组（忽略进出标记）
2. 在每个账户组内按时间排序
3. 自动识别：第一张为进，第二张为出，依此类推
4. 创建带账户ID的摆卡记录

**示例**：
```
输入数据：
08:00:00 - 无标记 (账户A)
08:05:00 - 无标记 (账户A)
09:00:00 - 无标记 (账户A)
09:05:00 - 无标记 (账户B)
10:00:00 - 无标记 (账户B)

处理结果：
账户A: [08:00:00, 08:05:00, 09:00:00]
  → 配对1: 账户A, 08:00:00(进) + 08:05:00(出)
  → 单卡1: 账户A, 09:00:00(进)

账户B: [09:05:00, 10:00:00]
  → 配对2: 账户B, 09:05:00(进) + 10:00:00(出)
```

#### 优先级3：进出标记成对

**触发条件**：
- 打卡记录无劳动力账户ID（accountId == null）
- 打卡记录有进出标记（punchType IN/OUT）
- 或者优先级1和2处理后的剩余记录

**处理步骤**：
1. 按进出标记分组（进/出）
2. 在进组和出组之间进行配对
3. 按时间顺序配对
4. 创建不带账户ID的摆卡记录（accountId = null）

**示例**：
```
输入数据：
08:00:00 - 进 (无账户)
08:05:00 - 进 (无账户)
09:00:00 - 出 (无账户)
09:05:00 - 出 (无账户)

处理结果：
进: [08:00:00, 08:05:00]
出: [09:00:00, 09:05:00]
  → 配对1: 无账户, 08:00:00(进) + 09:00:00(出)
  → 配对2: 无账户, 08:05:00(进) + 09:05:00(出)
```

#### 优先级4：相邻配对（第一笔进，第二笔出）

**触发条件**：
- 打卡记录无劳动力账户ID（accountId == null）
- 打卡记录无进出标记（punchType 为空或UNKNOWN）
- 或者优先级1、2、3处理后的剩余记录

**处理步骤**：
1. 合并所有剩余打卡记录
2. 按时间排序
3. 第一张视为进，第二张视为出，第三张视为进，第四张视为出，依此类推
4. 创建不带账户ID的摆卡记录（accountId = null）

**示例**：
```
输入数据：
08:00:00 - 无标记 (无账户)
08:05:00 - 无标记 (无账户)
09:00:00 - 无标记 (无账户)
09:05:00 - 无标记 (无账户)
10:00:00 - 无标记 (无账户)

处理结果：
所有记录: [08:00:00, 08:05:00, 09:00:00, 09:05:00, 10:00:00]
  → 配对1: 无账户, 08:00:00(进) + 08:05:00(出)
  → 配对2: 无账户, 09:00:00(进) + 09:05:00(出)
  → 单卡1: 无账户, 10:00:00(进) [奇数个，最后一笔单独处理]
```

## 典型异常场景

### 场景1：混合数据 - 部分有账户部分无账户

**输入数据**：
```
08:00:00 - 进 (账户A)      # 有账户+有标记
08:05:00 - 进 (账户B)      # 有账户+有标记
09:00:00 - 出 (无账户)     # 无账户+有标记
09:05:00 - 出 (账户A)      # 有账户+有标记
09:10:00 - 出 (账户B)      # 有账户+有标记
```

**处理流程**：
1. **优先级1处理**：
   - 账户A+进: [08:00:00]
   - 账户A+出: [09:05:00]
   - 账户B+进: [08:05:00]
   - 账户B+出: [09:10:00]
   - 配对1: 账户A, 08:00:00 + 09:05:00
   - 配对2: 账户B, 08:05:00 + 09:10:00

2. **剩余记录**：
   - 进: []
   - 出: [09:00:00] (无账户)

3. **优先级3处理**：
   - 单卡1: 无账户, 09:00:00 (只有签退)

**最终结果**：
- 配对1: 账户A, 08:00:00(进) + 09:05:00(出)
- 配对2: 账户B, 08:05:00(进) + 09:10:00(出)
- 单卡1: 无账户, 09:00:00(出)

### 场景2：混合数据 - 部分有标记部分无标记

**输入数据**：
```
08:00:00 - 进 (账户A)      # 有标记
08:05:00 - 无标记 (账户A)  # 无标记
09:00:00 - 出 (账户A)      # 有标记
09:05:00 - 无标记 (账户A)  # 无标记
```

**处理流程**：
1. **优先级1处理**：
   - 账户A+进: [08:00:00]
   - 账户A+出: [09:00:00]
   - 配对1: 账户A, 08:00:00 + 09:00:00

2. **剩余记录**：
   - 账户A: [08:05:00, 09:05:00] (无标记)

3. **优先级2处理**：
   - 配对2: 账户A, 08:05:00(进) + 09:05:00(出)

**最终结果**：
- 配对1: 账户A, 08:00:00(进) + 09:00:00(出) [优先级1]
- 配对2: 账户A, 08:05:00(进) + 09:05:00(出) [优先级2]

### 场景3：复杂混合 - 多种情况并存

**输入数据**：
```
08:00:00 - 进 (账户A)      # 有账户+有标记
08:05:00 - 进 (无账户)     # 无账户+有标记
08:10:00 - 无标记 (账户B)  # 有账户+无标记
08:15:00 - 无标记 (无账户) # 无账户+无标记
09:00:00 - 出 (账户A)      # 有账户+有标记
09:05:00 - 出 (无账户)     # 无账户+有标记
09:10:00 - 无标记 (账户B)  # 有账户+无标记
09:15:00 - 无标记 (无账户) # 无账户+无标记
```

**处理流程**：
1. **优先级1处理**（有账户+有标记）：
   - 账户A: 08:00:00(进) + 09:00:00(出)

2. **优先级2处理**（有账户+无标记）：
   - 账户B: 08:10:00(进) + 09:10:00(出)

3. **优先级3处理**（无账户+有标记）：
   - 无账户: 08:05:00(进) + 09:05:00(出)

4. **优先级4处理**（无账户+无标记）：
   - 无账户: 08:15:00(进) + 09:15:00(出)

**最终结果**：
- 配对1: 账户A, 08:00:00 + 09:00:00 [优先级1]
- 配对2: 账户B, 08:10:00 + 09:10:00 [优先级2]
- 配对3: 无账户, 08:05:00 + 09:05:00 [优先级3]
- 配对4: 无账户, 08:15:00 + 09:15:00 [优先级4]

### 场景4：极端异常 - 只有单一类型

**情况A：只有进卡**
```
输入数据：
08:00:00 - 进 (账户A)
08:05:00 - 进 (账户B)

处理结果：
- 单卡1: 账户A, 08:00:00(进)
- 单卡2: 账户B, 08:05:00(进)
```

**情况B：只有出卡**
```
输入数据：
09:00:00 - 出 (账户A)
09:05:00 - 出 (账户B)

处理结果：
- 单卡1: 账户A, 09:00:00(出)
- 单卡2: 账户B, 09:05:00(出)
```

**情况C：所有卡都无账户无标记**
```
输入数据：
08:00:00 - 无标记 (无账户)
08:05:00 - 无标记 (无账户)
09:00:00 - 无标记 (无账户)

处理结果：
- 配对1: 无账户, 08:00:00(进) + 08:05:00(出)
- 单卡1: 无账户, 09:00:00(进)
```

## 边界情况处理

### 边界情况1：奇数个打卡记录

**输入数据**：
```
08:00:00 - 进 (账户A)
08:05:00 - 进 (账户B)
09:00:00 - 出 (账户A)
```

**处理结果**：
- 配对1: 账户A, 08:00:00(进) + 09:00:00(出)
- 单卡1: 账户B, 08:05:00(进)

**说明**：最后一笔无法配对的打卡记录会被创建为单卡摆卡。

### 边界情况2：时间顺序异常

**输入数据**：
```
09:00:00 - 出 (账户A)      # 出现在前面
08:00:00 - 进 (账户A)      # 出现在后面
```

**处理结果**：
- 配对1: 账户A, 08:00:00(进) + 09:00:00(出)
  （系统会自动按时间排序后再配对）

**说明**：系统在处理前会自动按时间排序，确保正确配对。

### 边界情况3：时间间隔不满足要求

**配置**：摆卡间隔 = 5分钟

**输入数据**：
```
08:00:00 - 进 (账户A)
08:02:00 - 出 (账户A)      # 间隔只有2分钟
08:10:00 - 出 (账户A)      # 间隔10分钟
```

**处理结果**：
- 配对1: 账户A, 08:00:00(进) + 08:10:00(出)
  （08:02:00的出卡被跳过，因为不满足5分钟间隔要求）

**说明**：如果配置了摆卡间隔，系统会跳过不满足间隔要求的配对。

### 边界情况4：账户ID为null与字符串"null"

**说明**：系统会正确处理以下两种情况：
1. `accountId = null`（数据库中的NULL值）
2. `accountId = "null"`（字符串值，异常情况）

**处理**：
- `null` 和 `"null"` 都会被视为"无账户"
- 两者会被分到同一组进行处理

### 验证要点

#### 1. 优先级顺序正确性

✅ 优先级1（账户+标记）> 优先级2（账户）> 优先级3（标记）> 优先级4（相邻）

**验证方法**：
```
准备测试数据，包含4种情况的打卡记录
观察日志输出，确认优先级顺序
检查最终配对结果，确认高优先级先执行
```

#### 2. 账户ID处理正确性

✅ 有账户的配对带accountId，无账户的配对accountId为null

**验证方法**：
```
SELECT id, employeeNo, accountId, inPunchTime, outPunchTime
FROM PunchPair
WHERE employeeNo = 'TEST001'
  AND pairDate = '2025-01-15'
ORDER BY inPunchTime;
```

#### 3. 进出标记处理正确性

✅ 有标记的打卡使用原始punchType，无标记的自动识别

**验证方法**：
```
SELECT
  pp.id,
  pp.inPunchTime,
  pp.outPunchTime,
  in_pr.punchType as originalInType,
  out_pr.punchType as originalOutType
FROM PunchPair pp
LEFT JOIN PunchRecord in_pr ON pp.inPunchId = in_pr.id
LEFT JOIN PunchRecord out_pr ON pp.outPunchId = out_pr.id
WHERE pp.employeeNo = 'TEST001'
  AND pp.pairDate = '2025-01-15';
```

#### 4. 单卡处理正确性

✅ 只有进或只有出的打卡记录被正确处理为单卡

**验证方法**：
```
SELECT id, inPunchId, outPunchId, workHours
FROM PunchPair
WHERE employeeNo = 'TEST001'
  AND pairDate = '2025-01-15'
  AND (inPunchId IS NULL OR outPunchId IS NULL);
```

## 总结

异常情况处理的核心原则：

1. **优先级明确**：按照 优先级1 → 优先级2 → 优先级3 → 优先级4 的顺序处理
2. **数据完整**：确保所有打卡记录都被处理，不遗漏任何数据
3. **配对准确**：严格按照业务规则进行配对，确保准确性
4. **单卡处理**：对于无法配对的打卡记录，创建单卡摆卡
5. **日志详细**：输出详细的处理日志，便于排查问题

通过这种基于优先级的异常情况处理机制，系统能够灵活应对各种复杂的刷卡场景，确保摆卡数据的准确性和完整性。
