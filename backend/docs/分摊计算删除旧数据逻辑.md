# 分摊计算删除旧数据逻辑

## 修复时间
2026-03-31

## 问题描述
在执行分摊计算时，如果对同一配置多次执行（特别是配置规则发生变更），需要先删除当天历史的分摊结果，再重新生成最新的分摊结果。

### 实际场景
- **第一次分摊**：张三一天有8小时工时，配置A产生2条分摊结果
- **修改配置**：调整配置A的分摊规则或目标
- **第二次分摊**：配置A产生3条分摊结果
- **问题**：如果没有删除旧数据，数据库中会保留旧的2条结果，导致数据重复

## 修复方案

### 1. 优化删除逻辑

#### 原逻辑（有问题）
```javascript
// 循环删除，性能较低
const uniqueDates = Object.keys(groupedByDate);
for (const dateKey of uniqueDates) {
  const calcDate = new Date(dateKey);
  const deletedResults = await this.prisma.allocationResult.deleteMany({
    where: {
      configId: config.id,
      recordDate: calcDate,
    },
  });
}
```

#### 新逻辑（已修复）
```javascript
// 一次性批量删除，性能更高
const uniqueDates = Object.keys(groupedByDate);
const dateObjects = uniqueDates.map(dateKey => new Date(dateKey));

console.log(`[分摊计算] 开始清理旧分摊数据...`);
console.log(`  - 配置ID: ${config.id}`);
console.log(`  - 日期范围: ${uniqueDates.join(', ')}`);
console.log(`  - 日期数量: ${uniqueDates.length}`);

// 批量删除所有相关日期的旧分摊结果
const deletedResults = await this.prisma.allocationResult.deleteMany({
  where: {
    configId: config.id,
    recordDate: { in: dateObjects },
  },
});

console.log(`[分摊计算] 清理完成: 共删除 ${deletedResults.count} 条旧分摊结果`);

if (deletedResults.count > 0) {
  console.log(`[分摊计算] 旧数据已清理，将生成新的分摊结果`);
} else {
  console.log(`[分摊计算] 没有找到旧分摊结果，这是首次分摊`);
}
```

### 2. 删除范围说明

| 表名 | 是否删除 | 说明 |
|------|---------|------|
| `AllocationResult` | ✅ 删除 | 分摊结果记录（分摊关系） |
| `CalcResult` | ❌ 不删除 | 工时记录（源数据，必须保留） |
| `LaborAccount` | ❌ 不删除 | 账户信息 |
| `Employee` | ❌ 不删除 | 员工信息 |

**重要**：
- 只删除分摊结果（AllocationResult），不删除工时记录（CalcResult）
- 源数据（被分摊的原始工时）必须保留，作为审计依据
- 目标数据（分摊产生的工时）也保留，作为历史记录

### 3. 删除时机

```
分摊计算流程：
  1. 查询待分摊的工时记录
  2. 按日期分组
  3. 【删除旧分摊结果】← 在这里执行
  4. 生成新的分摊结果
  5. 保存分摊结果
  6. 记录审计日志
```

**关键点**：
- 在生成新数据之前完成删除
- 确保删除操作在单个事务中
- 删除完成后才开始生成新数据

### 4. 添加完成日志

```javascript
console.log(`[分摊计算] ========== 分摊计算完成 ==========`);
console.log(`  - 处理日期数: ${uniqueDates.length}`);
console.log(`  - 生成分摊结果: ${resultCount} 条`);
console.log(`  - 批次号: ${batchNo}`);
console.log(`[分摊计算] =====================================`);
```

## 数据一致性保证

### 1. 原子性
- 删除操作在单个数据库事务中执行
- 要么全部成功，要么全部回滚

### 2. 完整性
- 删除所有旧数据后才开始生成新数据
- 避免新旧数据共存导致的数据不一致

### 3. 可追溯性
- 记录详细的删除日志
- 包含配置ID、日期范围、删除条数

### 4. 性能优化
- 批量删除比循环删除效率更高
- 减少数据库连接开销

## 注意事项

1. **不要删除工时记录**：`CalcResult` 表的工时记录是源数据，绝对不能删除
2. **按配置和日期删除**：只删除指定配置和指定日期的结果，不影响其他配置或日期
3. **软删除机制**：表中有 `deletedAt` 字段，建议使用软删除而不是硬删除
4. **批次号追踪**：每次生成分摊结果都会生成新的批次号，便于追踪
