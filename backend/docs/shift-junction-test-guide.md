# 班次连接点跨越打卡处理测试指南

## 概述

本文档用于验证班次连接点跨越打卡的处理逻辑。当两个班次连在一起时，系统会自动处理以下两种情况：

1. **跨班次配对**：员工在前一个班次进，在后一个班次出，系统自动在连接点插入虚拟打卡
2. **连接点附近打卡**：打卡记录在连接点±1分钟范围内，系统自动处理归属

## 业务场景

### 连接点定义

**连接点**：两个班次之间的中间时间点

**判断条件**：
- 下一个班次开始时间 <= 当前班次结束时间 + 1分钟
- 连接点时间 = (当前班次结束时间 + 下一个班次开始时间) / 2

**处理规则**：
- 开始时间 ~ 连接点 → 归属前一个班次
- 连接点 ~ 结束时间 → 归属后一个班次

### 虚拟打卡记录

当检测到跨班次配对时，系统会自动创建两条虚拟打卡记录：

1. **虚拟出卡**（时间点出）
   - 时间：连接点时间
   - 类型：OUT
   - 归属：前一个班次
   - 标记：`VIRTUAL_JUNCTION_OUT_{前班次ID}_{后班次ID}`

2. **虚拟进卡**（时间点进）
   - 时间：连接点时间
   - 类型：IN
   - 归属：后一个班次
   - 标记：`VIRTUAL_JUNCTION_IN_{前班次ID}_{后班次ID}`

## 业务场景

## 测试场景

### 场景0：跨班次配对（主要场景）

**排班配置**：
```
班次A：08:00 - 12:00
班次B：12:00 - 17:00
连接点：12:00
```

**打卡数据**：
```
08:00:00 - 进 (设备1)
14:00:00 - 出 (设备1)  # 跨越两个班次
```

**预期结果**：
```
1. 检测到连接点：12:00
2. 检测到跨班次配对：
   - 班次A有进卡（08:00），无出卡
   - 班次B有出卡（14:00），无进卡
3. 插入虚拟打卡：
   - 12:00:00 - OUT (虚拟，归属班次A)
   - 12:00:00 - IN (虚拟，归属班次B)
4. 最终配对：
   班次A：08:00:00(IN) + 12:00:00(OUT虚拟) → 工时：4小时
   班次B：12:00:00(IN虚拟) + 14:00:00(OUT) → 工时：2小时
```

**验证SQL**：
```sql
-- 查看虚拟打卡记录
SELECT
  id,
  employeeNo,
  punchTime,
  punchType,
  source
FROM PunchRecord
WHERE employeeNo = 'TEST001'
  AND source LIKE 'VIRTUAL_JUNCTION%'
  AND punchTime BETWEEN '2025-01-15 00:00:00' AND '2025-01-15 23:59:59'
ORDER BY punchTime;

-- 预期结果：
-- 虚拟出卡：12:00:00, OUT, VIRTUAL_JUNCTION_OUT_班次AID_班次BID
-- 虚拟进卡：12:00:00, IN, VIRTUAL_JUNCTION_IN_班次AID_班次BID

-- 查看摆卡结果
SELECT
  pp.id,
  pp.shiftName,
  pp.inPunchTime,
  pp.outPunchTime,
  pp.workHours,
  in_pr.source as inPunchSource,
  out_pr.source as outPunchSource
FROM PunchPair pp
LEFT JOIN PunchRecord in_pr ON pp.inPunchId = in_pr.id
LEFT JOIN PunchRecord out_pr ON pp.outPunchId = out_pr.id
WHERE pp.employeeNo = 'TEST001'
  AND pp.pairDate = '2025-01-15'
ORDER BY pp.inPunchTime;

-- 预期结果：
-- 班次A：08:00:00 + 12:00:00(虚拟) → 4小时
-- 班次B：12:00:00(虚拟) + 14:00:00 → 2小时
```

### 场景1：标准的班次连接（紧密相连）

**排班配置**：
```
班次A：08:00 - 12:00
班次B：12:00 - 17:00
连接点：(12:00 + 12:00) / 2 = 12:00
```

**打卡数据**：
```
11:58:00 - 进 (设备1)
12:02:00 - 出 (设备1)  # 跨越连接点 12:00
```

**预期结果**：
```
1. 检测到连接点：12:00
2. 检测到跨越打卡：12:02:00 (在连接点±1分钟范围内)
3. 插入虚拟打卡：
   - 12:00:00 - OUT (虚拟，归属班次A)
   - 12:00:00 - IN (虚拟，归属班次B)
4. 最终配对：
   班次A：11:58:00(IN) + 12:00:00(OUT虚拟)
   班次B：12:00:00(IN虚拟) + 12:02:00(OUT)
```

**验证SQL**：
```sql
-- 查看虚拟打卡记录
SELECT
  id,
  employeeNo,
  punchTime,
  punchType,
  source
FROM PunchRecord
WHERE employeeNo = 'TEST001'
  AND source LIKE 'VIRTUAL_JUNCTION%'
  AND punchTime BETWEEN '2025-01-15 00:00:00' AND '2025-01-15 23:59:59'
ORDER BY punchTime;

-- 预期结果：
-- 虚拟出卡：12:00:00, OUT, VIRTUAL_JUNCTION_OUT_班次AID_班次BID
-- 虚拟进卡：12:00:00, IN, VIRTUAL_JUNCTION_IN_班次AID_班次BID
```

### 场景2：班次重叠（下一个班次提前开始）

**排班配置**：
```
班次A：08:00 - 12:00
班次B：11:30 - 17:00  # 提前开始
连接点：(12:00 + 11:30) / 2 = 11:45
```

**打卡数据**：
```
11:44:00 - 进 (设备1)
11:46:00 - 出 (设备1)  # 跨越连接点 11:45
```

**预期结果**：
```
1. 检测到连接点：11:45
2. 检测到跨越打卡：11:46:00 (在连接点±1分钟范围内)
3. 插入虚拟打卡：
   - 11:45:00 - OUT (虚拟，归属班次A)
   - 11:45:00 - IN (虚拟，归属班次B)
4. 最终配对：
   班次A：11:44:00(IN) + 11:45:00(OUT虚拟)
   班次B：11:45:00(IN虚拟) + 11:46:00(OUT)
```

### 场景3：打卡记录正好在连接点上

**排班配置**：
```
班次A：08:00 - 12:00
班次B：12:00 - 17:00
连接点：12:00
```

**打卡数据**：
```
12:00:00 - 进 (设备1)  # 正好在连接点
```

**预期结果**：
```
1. 检测到连接点：12:00
2. 检测到跨越打卡：12:00:00 (在连接点±1分钟范围内)
3. 插入虚拟打卡：
   - 12:00:00 - OUT (虚拟，归属班次A)
   - 12:00:00 - IN (虚拟，归属班次B)
4. 最终配对：
   班次A：无进卡（原始12:00:00的IN被归属到班次B）
   班次B：12:00:00(IN虚拟) + 12:00:00(IN) → 配对失败（两张都是进卡）
```

**注意**：这种情况可能产生单卡摆卡。

### 场景4：多天班次跨越（夜班）

**排班配置**：
```
第一天夜班：22:00 - 06:00 (次日)
第二天白班：06:00 - 14:00
连接点：06:00
```

**打卡数据**：
```
05:59:00 - 进 (设备1)
06:01:00 - 出 (设备1)  # 跨越连接点 06:00
```

**预期结果**：
```
1. 检测到连接点：06:00
2. 检测到跨越打卡：06:01:00 (在连接点±1分钟范围内)
3. 插入虚拟打卡：
   - 06:00:00 - OUT (虚拟，归属第一天夜班)
   - 06:00:00 - IN (虚拟，归属第二天白班)
4. 最终配对：
   第一天夜班：05:59:00(IN) + 06:00:00(OUT虚拟)
   第二天白班：06:00:00(IN虚拟) + 06:01:00(OUT)
```

### 场景5：多个班次连续相连

**排班配置**：
```
班次A：08:00 - 12:00
班次B：12:00 - 17:00
班次C：17:00 - 22:00
连接点1：12:00 (班次A和B之间)
连接点2：17:15 (班次B和C之间，(17:00+17:30)/2)
```

**打卡数据**：
```
11:58:00 - 进 (设备1)
12:02:00 - 出 (设备1)  # 跨越连接点1
16:58:00 - 进 (设备1)
17:02:00 - 出 (设备1)  # 跨越连接点2
```

**预期结果**：
```
1. 检测到2个连接点
2. 为每个连接点插入虚拟打卡
3. 最终配对：
   班次A：11:58:00(IN) + 12:00:00(OUT虚拟)
   班次B：12:00:00(IN虚拟) + 12:02:00(OUT) + 16:58:00(IN) + 17:15:00(OUT虚拟)
   班次C：17:15:00(IN虚拟) + 17:02:00(OUT)
```

## 边界情况

### 边界情况1：打卡记录不在连接点附近

**打卡数据**：
```
08:00:00 - 进 (设备1)
11:00:00 - 出 (设备1)  # 远离连接点 12:00
13:00:00 - 进 (设备1)
17:00:00 - 出 (设备1)
```

**预期结果**：
```
- 不插入虚拟打卡记录
- 正常配对：
  班次A：08:00:00(IN) + 11:00:00(OUT)
  班次B：13:00:00(IN) + 17:00:00(OUT)
```

### 边界情况2：只有一个班次

**排班配置**：
```
只有班次A：08:00 - 12:00
没有班次B
```

**预期结果**：
```
- 不检测连接点（少于2个班次）
- 正常处理，不插入虚拟打卡
```

### 边界情况3：打卡记录时间相同

**打卡数据**：
```
12:00:00 - 进 (设备1)
12:00:00 - 出 (设备1)  # 完全相同的时间
```

**预期结果**：
```
- 两条打卡记录都被认为是跨越连接点
- 插入虚拟打卡：
  - 12:00:00 - OUT (虚拟)
  - 12:00:00 - IN (虚拟)
- 最终可能有4条打卡记录（2条原始 + 2条虚拟）
```

### 边界情况4：连接点判断阈值

**测试不同的班次间隔**：

```
班次A结束：12:00
班次B开始：12:00  → 连接点：12:00，阈值：12:01 → 有连接点
班次A结束：12:00
班次B开始：12:01  → 连接点：12:00:30，阈值：12:01 → 有连接点
班次A结束：12:00
班次B开始：12:02  → 连接点：12:01，阈值：12:01 → 有连接点
班次A结束：12:00
班次B开始：12:03  → 连接点：12:01:30，阈值：12:01 → 无连接点
```

**结论**：只有当班次B开始时间 <= 班次A结束时间 + 1分钟时，才认为有连接点。

## 性能影响

### 额外处理

1. **连接点检测**：O(n)，n为排班数量
2. **跨越打卡检测**：O(m×k)，m为打卡记录数，k为连接点数量
3. **虚拟打卡创建**：O(2k)，k为跨越打卡数量
4. **打卡记录分配**：O(m×n)，m为打卡记录数，n为班次数

### 总体性能

- 单个员工摆卡：预计增加 10-20ms
- 批量摆卡（100人）：预计增加 1-2秒

## 注意事项

### 1. 虚拟打卡记录的识别

虚拟打卡记录通过 `source` 字段标识：
- 格式：`VIRTUAL_JUNCTION_OUT_{前班次ID}_{后班次ID}`
- 格式：`VIRTUAL_JUNCTION_IN_{前班次ID}_{后班次ID}`

这些虚拟打卡记录：
- 不会同步到考勤机
- 不会被删除（除非手动删除摆卡记录）
- 在日志中会有特殊标识

### 2. 工时计算

虚拟打卡记录参与工时计算：
- 班次A：计算到连接点的工时
- 班次B：计算从连接点开始的工时

**示例**：
```
班次A：11:58(IN) + 12:00(OUT虚拟) → 工时：2分钟
班次B：12:00(IN虚拟) + 12:02(OUT) → 工时：2分钟
```

### 3. 摆卡记录删除

当重新摆卡时：
- 删除所有旧的摆卡记录（包括虚拟打卡创建的）
- 删除所有旧的虚拟打卡记录
- 重新检测连接点和跨越打卡
- 重新创建虚拟打卡和摆卡记录

### 4. 手动干预

如果自动处理的连接点不正确，管理员可以：
1. 删除虚拟打卡记录
2. 删除相关摆卡记录
3. 手动创建正确的摆卡记录
4. 或调整排班配置，避免班次相连
