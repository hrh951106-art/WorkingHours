## A06分摊规则执行失败原因分析报告

### 问题现象
执行A06间接工时分摊规则后，没有产生任何分摊数据。

### 日志分析

从后端日志中发现以下关键信息：

---

### ❌ 问题1：层级索引映射错误

**日志显示**：
```
源账户路径: DH/DH01/DH01001/A01/-/FULL_TIME/FINANCE
路径分段: [DH, DH01, DH01001, A01, -, FULL_TIME, FINANCE]
分摊范围组织: W1总装车间 (代码: DH01, 类型: 03)
分摊范围层级: 2 (类型03)
提取的层级值: DH01001  ← ❌ 错误！这是产线代码，不是车间代码
```

**问题分析**：

账户路径结构（按 `/` 分隔）：
- 索引0: `DH` - 工厂代码
- 索引1: `DH01` - 车间代码 ✅ **应该是这个**
- 索引2: `DH01001` - 产线代码 ❌ 当前代码错误地使用了这个
- 索引3: `A01` - 产品代码
- 索引4: `-` - 占位符
- 索引5: `FULL_TIME` - 聘用形式
- 索引6: `FINANCE` - 岗位

**当前代码的层级映射**：
```typescript
const typeLevelMap: Record<string, number> = {
  '01': 0, // 集团
  '02': 1, // 工厂
  '03': 2, // 车间 ← 错误！应该是1
  '04': 3, // 产线 ← 错误！应该是2
};
```

**错误的映射导致**：
- 分摊范围是车间（类型03）
- 代码使用索引2，提取到的是 `DH01001`（产线）
- 开线记录的账户路径第2级是 `DH01`（车间）
- 匹配失败：期望 `DH01001`，实际是 `DH01`

**正确的映射应该是**：
```typescript
const typeLevelMap: Record<string, number> = {
  '01': 0, // 集团 - 索引0（如果有）
  '02': 0, // 工厂 - 索引0
  '03': 1, // 车间 - 索引1 ✅
  '04': 2, // 产线 - 索引2 ✅
};
```

---

### ❌ 问题2：即使匹配成功，直接工时为0

**日志显示**：
```
步骤4: 确定分摊目标
  共匹配到 1 个分摊目标
  目标1: 组织ID=7, 名称=W1总装车间L1产线

分摊范围总直接工时: 0
❌ 分摊范围的直接工时为0，无法按比例分摊，跳过分摊
```

**问题分析**：

即使匹配到了1个分摊目标（组织ID=7），但是该目标的直接工时为0。

这是因为：
1. 在查询直接工时时，使用的是 `组织ID=7`（W1总装车间L1产线）
2. 但是系统中没有该产线在 2026-05-14 班次ID=3 的直接工时记录
3. 或者直接工时记录的 `actualHoursAttendanceCodeId` 不匹配

---

### 根本原因总结

#### 主要原因：层级索引映射错误

**错误的逻辑**：
```
账户路径: DH/DH01/DH01001/A01/...
          ↓ 类型03 (车间) → 索引2
提取层级值: DH01001 (产线代码) ❌
```

**应该的逻辑**：
```
账户路径: DH/DH01/DH01001/A01/...
          ↓ 类型03 (车间) → 索引1
提取层级值: DH01 (车间代码) ✅
```

#### 次要原因：直接工时数据缺失

即使修复了层级映射，还需要确保：
1. 匹配到的产线有直接工时记录（A02_LINE）
2. 直接工时的日期和班次与待分摊数据一致
3. 直接工时的账户匹配正确

### 结论

**A06规则没有产生分摊数据的根本原因**：

1. **层级索引映射错误**（主要原因）
   - 类型03（车间）映射到了索引2（产线）
   - 应该映射到索引1（车间）
   - 导致提取的层级值不正确，无法匹配开线记录

2. **直接工时为0**（次要原因）
   - 即使偶然匹配到1个目标，该目标也没有直接工时
   - 导致无法按比例计算分摊

**需要修复的位置**：
- 文件：`src/modules/allocation/allocation.service.ts`
- 方法：`executeActualHoursAllocation`
- 行数：约1909行
- 代码：`typeLevelMap` 映射表

### 正确的层级映射关系

│   └─ 车间代码 (索引1，类型03) ✅
